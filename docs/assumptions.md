# 需求理解与假设说明

## 关键留白点

1. 分级审批金额阈值  
   假设 0-500 元为一级审批，500 元以上进入二级审批。理由：低额货损可由仓库主管快速处理，高额赔付/追偿涉及财务和供应商争议，需要二级复核。系统中通过 `v3_approval_rules` 配置，不写死在代码分支里。

2. 审批超时时长  
   一级审批 8 小时，二级审批 16 小时。理由：一级处理多为当天班次内完成，二级审批通常涉及跨部门确认，允许更长窗口。后台任务 `/api/jobs/run` 自动升级或关闭。

3. 重提次数上限  
   拒绝后最多重提 2 次。理由：允许补充材料，但避免同一工单无限循环占用审批资源。超过上限后，一级拒绝可升级二级，二级拒绝自动关闭。

4. 物流异常类型映射  
   丢件：客户理赔 + 重新发货 + 原库存出库追踪。  
   破损：客户理赔 + 重新发货。  
   客户拒收：退货入库，通常不赔付。  
   超时未签收：先追踪，必要时客户理赔。  
   地址错误：重新发货或改派，默认不赔付。  
   理由：不同异常对客户承诺、库存和财务影响不同，不能全部按同一种赔付动作处理。

5. 角色权限划分  
   操作员可手工上报和扫描；一级审批人处理一级审批；二级审批人处理二级审批；品控主管可扫描并执行误判快速放行；管理员可兜底处理。上报人不能审批自己提交的工单，后端接口强制校验。

6. V2 数据同步频率与一致性策略  
   关键动作实时拉取 V2：手工上报和扫描前都调用 V2 校验。列表/详情使用本地快照展示，同时标注数据来源与同步时间。建议生产环境每 10 分钟增量同步一次，夜间跑全量对账。若 V2 运单金额在异常处理期间变化，V3 记录版本差异并要求审批人在详情页确认，以审批发生时的 V3 工单金额为赔付依据。

7. 品控暂扣超时时长  
   设定为 90 分钟，独立于审批超时。理由：品控暂扣会造成批次压仓和出库阻塞，成本高于普通审批等待，应远短于 8/16 小时审批窗口。超时后强制升级二级审批，不直接放行。

8. 品控规则触发阈值  
   数量差异超过 3% 触发数量不符；外观破损等级 >= 2 触发外观破损；规格偏差 > 5mm 触发规格不符；标签不可识别触发标签错误；批次库龄可扩展为超过 30 天触发批次异常。理由：数量差异和破损等级直接影响发货准确率，标签/规格问题会引起收货和售后风险。规则存放在 `v3_quality_rules`，后台可调整；若规则关闭 `autoCreateTicket`，系统只记录异常扫描，等待人工复核，不自动锁定批次或创建工单。

9. 品控主管角色权限边界  
   品控主管可以执行误判快速放行，必须填写复核原因并留痕；不能绕过二级审批处理真实质量问题；可以兼任操作员扫描职责，但默认不兼任一级/二级审批。若一人多角色，后端仍按当前操作所需权限和自批自核规则校验。

## 其他工程假设

- 当前演示按单租户为主，但表结构已包含 `tenantId` 和 `warehouseId`。若多租户上线，所有查询都必须带租户和仓库条件，V2 API Key 也应绑定租户权限。
- 同一运单同类型未关闭物流工单不允许重复上报。品控工单未关闭期间允许另一个物流异常并存，因为发货前质量问题和发货后物流问题属于不同责任链路。
- 赔付记录统一表存储，但通过 `direction` 区分“赔付给客户”和“向供应商追偿”，对账方式也不同。
- 审批通过与库存/赔付联动在 `writeState` 事务中完成；使用 Postgres 时加表锁并一次性保存，内存演示环境也保持同一临界区语义。
- 幂等性通过工单当前状态、版本号和审批 `idempotencyKey` 共同保证。

## 想向产品经理确认的问题

- 高额异常是否还需要财务岗或法务岗三审？
- 品控快速放行是否需要上传图片证据？
- 地址错误由客户造成和由仓库录入造成时，赔付责任是否不同？
- V2 是否已有异常状态字段，回写后是否会影响旧报表？
- 供应商追偿是否需要拆分采购单、收货批次和质检报告三方关联？

## 反思题

1. 如果题目直接给出精确阈值和超时时长，架构不会改为硬编码，仍会保留规则表。区别只是默认种子值来自题目，而不是假设文档。

2. 最大风险是审批状态已经变成通过，但库存扣减或赔付生成失败。当前用事务保证一致；如果不用事务，会采用 Outbox/Saga：先记录待执行事件，后台幂等消费，失败可重试并显示补偿状态。

3. 面对一句话模糊需求，我倾向于先问关键问题，同时做一个最小可运行版本。纯等待会拖慢反馈，纯动手容易把错误假设固化。

4. 新增“客户投诉服务态度”时，可新增异常类型和动作映射，设为不联动库存、不生成赔付或仅生成客服处理记录。当前表结构不需要大改，状态机可复用。

5. 工单量到 20 万条时，最先撑不住的是列表查询和后台任务全表扫描。需要给状态、租户、运单号、审批人、到期时间建索引；后台任务改为按 `due_at` 分片拉取；接口同步改队列化。

6. 给 V2 新增接口时，使用版本化接口和字段兼容策略，不破坏旧调用方。金额从 int 到 decimal 时，V3 适配层兼容两种格式并统一转换为 cents，业务层不直接读取 V2 原始金额类型。
